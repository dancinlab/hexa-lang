────────────────────────────────────────────────────────────────────────────
KEY ASSESSMENT — byte-identical-vs-clang: bootstrapping convenience, NOT mandate
────────────────────────────────────────────────────────────────────────────
1. Is byte-identical-vs-clang governance-mandated?
   Grep of @D directives in project.tape, AGENTS.tape (empty), and
   RUNTIME.floor.md text: NO @D rule mandates byte-identity vs clang. The
   only "byte" reference in project.tape is paper_significance "verify /
   byte-diff / RUNEQ" — one of THREE allowed verification methods.

2. What ARE the real F3 invariants?
   (a) default 0-extern build stays byte-identical to origin/main
       (preserved here: shadow emitter, no `use`, no runtime.c edit;
        the existing #ifdef HEXA_RT_SELFEMIT guard is OPT-IN by design).
   (b) self-host fixpoint f(C)=C (tool/meta2_verify.hexa, v2==v3==v4 SHA
       equality) — the standing correctness anchor.
   (c) self-emitted runtime is FUNCTIONALLY correct (behaves like its C
       counterpart at the ABI level).
   None of (a)/(b)/(c) requires byte-identity to clang's output.

3. Where did byte-identical-vs-clang come from?
   The leaf-phase emitters (memset, strcmp, …) were small enough to match
   `as -arch arm64` of the C source character-for-character, so the
   verification pattern WAS byte-eq + JIT-exec. When class-B HARD bodies
   started requiring clang's optimizer choices (auto-vectorized copy, RA
   spill scheduling, csel fusion), this pattern silently inverted from
   "we COULD match" into "we MUST match," forcing the impossibility.

4. Behavioral-equivalence is the strictly correct gate.
   A self-hosting compiler emits ITS OWN correct machine code; correctness
   is observable behavior at the ABI boundary, not a textual diff vs a
   third-party optimizer. This is what every real bootstrapped compiler
   does (gcc, clang, rustc, ghc).

Governance verdict: NO rule mandates byte-identity vs clang. Behavioral-
equivalence (JIT-exec battery + gen1≡gen2 fixpoint + default-0-extern
byte-identity) is a STRICT SUFFICIENT correctness gate. Wall reframes:
"impossible" → "tractable engineering."

────────────────────────────────────────────────────────────────────────────
WHY THIS IS THE STRATEGIC RESULT — the wall is the wrong gate
────────────────────────────────────────────────────────────────────────────
The B9.6-E2 verdict measured 5 missing codegen capabilities (W1 csel/ccmp,
W2 multi-distinct-global reloc, W3 RA multi-pair spill, W4 interleaved bl,
W5 multi-BB merges) and concluded: TERMINAL WALL because byte-identical-vs-
clang requires reproducing those choices verbatim. This PR shows:

  * W3 (multi-pair frame: x29/x30 + x19/x20 + x21) — IMPLEMENTED (a fixed
    callee-saved layout, not RA-emergent, but ABI-correct: x19/x20/x21
    are stack-saved + restored, surviving the bl). The wall conflated
    "matching clang's RA" with "having a correct frame."

  * W4 (bl interleaved with other state) — IMPLEMENTED. The bl sits in the
    middle of the body, with computed-state setup (x0 = n+1) before and
    cbz + state-store + multi-loop after.

  * W5 (multi-BB merges: forward + back-edge to shared epilogue) —
    IMPLEMENTED. The .Lnull label is a shared epilogue reached by both the
    `cbz x0, .Lnull` early-out AND the `cbz x0, .Lnull` malloc-fail branch;
    .Lret is a join from the normal-return `b .Lret` AND the .Lnull fall-
    through. Two forward + one back-edge, exactly as B9.6-E2 enumerated.

  * W1 (csel/ccmp) and W2 (multi-distinct-global reloc) are NOT used here —
    because they are clang's CHOICES, not strdup's requirement. A scalar
    branching strdup needs neither.

The wall was: "to be byte-identical to clang, we need W1-W5 in clang's
exact form." The reframe is: "to be a correct self-emitted strdup, we need
W3/W4/W5 in OUR form, and that we have." Class-B HARD copy-loop primitives
(strdup, strndup, and ~60-100 sibling fn) are now TRACTABLE per-primitive
PORT, not "unbuilt codegen."

HONEST RESIDUAL — what this PR does NOT claim
────────────────────────────────────────────────────────────────────────────
- This is INCREMENT 1 of the reframe. It does NOT activate `_hxlcl_strdup`
  in runtime.c (no Path-A guard added). That is the next expert increment:
  add the `#ifdef HEXA_RT_SELFEMIT extern` guard around hxlcl_strdup +
  the callee-export wrinkle on hxlcl_malloc (same pattern as rt_atoi →
  hxlcl_atoll), wire it into build_hexa_cli.hexa's leaf emit loop, and
  do the dual-build 3-mode test. ~0.5-1 day, mechanical from here.

- This does NOT dissolve the alloc-core terminal (B9.6-E2). The bump
  allocator SEED is still irreducible (B9.8). What this PR shows is that
  the LAYER ABOVE the seed — class-B HARD copy-loop primitives — is
  tractable, which restores ~60-100 primitives to the F3 PORT backlog.

- `.c` count unchanged. The runtime.c removal is a downstream
  consequence of activating enough self-emitted primitives, not this PR.

- This does NOT rebuild the self-host fixpoint (heavy regen, would route
  to mini). The fixpoint stays intact because nothing `use`s the shadow
  emitter; the hexa_cc.c regen surface is untouched.
