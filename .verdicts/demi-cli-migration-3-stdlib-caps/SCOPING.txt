CAP (1) — KEYCHAIN CREDENTIAL STORE
═══════════════════════════════════════════════════════════════════════════════
EXISTS (before this cycle): NOTHING. No hexa stdlib keychain/secret/SecItem
  surface. (The `secret` skill / ~/.local/bin/secret is a bash CLI, not hexa.)
GAP: a get/set/delete credential surface callable from hexa.
HANDOFF FRAMING vs REALITY: handoff says "SecItemAdd/SecItemCopyMatching FFI".
  That OVERSTATES it. hexa HAS C FFI (stdlib/c_ffi.hexa: extern fn + @link
  framework) so SecItem CAN be bound — BUT it needs CFDictionary/CFString/
  CFTypeRef marshalling (a multi-call CoreFoundation arc). The canonical g1
  hexa-native path is a shellout to the base-installed macOS `security` CLI —
  the SAME pattern stdlib/http.hexa (curl) and stdlib/proc.hexa (shell) use, and
  exactly what the existing `secret` bash CLI does:
    security {find,add,delete}-generic-password -a <key> -s <service> [-w] ...
  NO new C FFI intrinsic, NO 4-surface registration (codegen/_is_builtin/
  runtime_core.c/bind.hexa) is required — those would only be needed for a raw
  SecItem extern-fn path, which buys nothing over the CLI for generic-password.
SIZE: 1 PR. *** SMALLEST / most self-contained → FIRST INCREMENT (LANDED). ***
DESIGN: stdlib/keychain.hexa — keychain_get / keychain_get_status / keychain_set
  / keychain_delete / keychain_has / keychain_available + pure command-builders
  (keychain_build_{get,set,delete}_cmd) for golden-fixture testing.
