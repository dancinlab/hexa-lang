ALGORITHM (the FA-2 formula, as in Dao 2023):
  Online softmax m_i, l_i carried ACROSS all K/V tiles (not per-tile reset)
  O accumulator REGISTER-RESIDENT across the entire K/V loop (NO smem o_tile
    and NO per-tile rescale store-load round-trip)
  S, P resident in smem only for the wmma-load-A handoff (S 1024 B, P 512 B)
  ONE kernel launch per (Q-block, head) streaming through all K/V tiles
  Inner GEMMs via wmma::mma_sync m16n16k16 fp16 fp32-acc

KEY FIX vs ROUND-4'S FAILED reg-O ATTEMPT
  Round-4 result.json noted: "register_resident_O: FAILED numeric (wrong
  m16n16k16 fp32 C-frag row map) -> dropped". Root cause empirically resolved
  via frag_probe.cu (in this fire dir): the lane-element-to-row map for the
  m16n16k16 f32 ACCUMULATOR fragment is
    elems {0,1,4,5} -> row = lane/4
    elems {2,3,6,7} -> row = lane/4 + 8
  Round-4 oracle used {0..3} -> lo and {4..7} -> hi (matches matrix_a/B maps
  but NOT the accumulator). Frag_probe.cu nails the actual layout by tag-store
  inspection, fixing the FAILED case.

WHAT THE GAP STILL IS (binding constraint, measured)
  At d=64, the per-row softmax + smem round-trip for S/P is now the dominant
  cost (no longer the O round-trip, which reg-O eliminated). To approach
  cuBLAS's wall further requires keeping S in REGISTERS across softmax via
  warp-shuffle reductions, and feeding P directly to the next mma's A
  fragment via mma.x4-from-regs (no smem P round-trip). This is a substantial
  rewrite of the inner softmax block and was OUT OF SCOPE for round 8.

FINDING (closed-negative-on-recipe-axis + positive-gap-closure on algorithm-axis)
  Register-resident O is the largest single algorithmic lever between the
  round-4 baseline and the FA-2 ceiling at d=64 -- it closes the gap from
  3.4-5.0x to 1.60-1.99x in ONE change (with a corrected fragment row map).
  The remaining 1.60-1.99x gap to cuBLAS-stack at d=64 is bound by the smem
  round-trip on S/P, not by the GEMM payload itself. cuBLAS's 3-launch
  HBM-materializing stack still beats the fused kernel on small head-dim
  because at d=64 the GEMM payload is too small to amortize launch+smem
  overhead. The dual-axis verdict framing remains: cuBLAS is NOT the ceiling
  on this fusable axis -- it just hasn't been exceeded YET. Open: register-S
  + warp-shuffle softmax rewrite (next round).
