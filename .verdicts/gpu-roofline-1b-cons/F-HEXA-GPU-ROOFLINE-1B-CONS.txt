WHY HONEST-STOP — the minimal-consumer premise does not hold (KEY FINDING)
================================================================================
The task's candidate minimal consumer (a fixed-size C array struct field `T name[N]` from
codegen's struct emitter) was based on `self/codegen_c.hexa::gen_struct_decl_c`, which DOES
emit a real C `typedef struct { long <field>; }`. BUT that 526-line backend is NOT the
transpiler the `hexa cc --regen` / hexa_v2 pipeline uses. The cc module manifest
(`self/lib.hexa`: lexer · parser · type_checker · codegen) names `codegen.hexa` (13327 lines),
NOT codegen_c.hexa.

The REAL backend (codegen.hexa) does NOT emit a typed C struct at all. `struct S {buf:[int;4]}`
lowers to a TYPE-ERASED HexaVal constructor (see _evidence/struct_lowering_sample.c, verbatim):
      HexaVal S(HexaVal buf) {
          static const char* const _k[] = {"buf"};
          HexaVal _v[] = {buf};
          return hexa_make_struct_n("S", 1, _k, _v);
      }
Only field NAMES survive (as runtime string keys); field TYPES — including the `[T; N]`
string the parser now carries — are DROPPED at struct lowering. There is no typed C struct
field for an `[N]` dimension to attach to. (codegen.hexa make_struct_n emit sites: L8538,
L8842.)

Consequence: making the captured N "change the emit-C" in the real transpiler is NOT a
<200-line additive single-consumer wire. It would require either (i) a new typed-struct
lowering path (typed C struct + typed field access + a typed-array runtime repr) or
(ii) threading N through HIR/MIR into the array-literal / struct-construction runtime calls —
both spanning parser AST + codegen + runtime across multiple modules = the multi-session
codegen scope @L4 designates for honest-STOP. Per @L4: "1b-cons -> [~] only if real consumer
landed w/ passing persisted gates; else keep [ ] + honest scoping." No real consumer landed,
